# 接口自动化测试平台 AI 模块重构实施计划书

## 1. 项目背景与现状问题

### 1.1 项目背景

本项目为 **前后端分离接口自动化测试平台**，技术栈：

- 前端：Vue
- 后端：Java + SpringBoot
- 执行引擎：Python + Unittest
- AI 服务：Python + FastAPI
- 数据库：MySQL
- 权限：RBAC + 项目隔离

平台对标 Apifox，支持接口管理、用例管理、任务调度，和新增待完善的AI 问答与用例生成。

### 1.2 现存核心问题

1. **向量库架构错误**：Chroma 按项目多文件夹隔离，无法并发、无法统一管理、无法高效删改
2. **Agent 逻辑过重**：使用 ReAct 循环自主决策、自主查接口、自主生成接口，不可控、格式不稳定
3. **无格式强约束**：无 JSON Schema、无 Pydantic 校验，AI 输出无法直接入库
4. **对话历史冗余**：同时使用前端 localStorage + 后端对话 ID，逻辑冲突
5. **用例生成链路混乱**：生成即入库、中间态多、异常难以处理
6. **RAG 检索无隔离**：未使用元数据过滤，存在项目数据泄露风险
7. **文档与向量无法联动删改**：删除业务文档无法同步删除向量

## 2. 重构总体目标

1. **稳定**：AI 输出 100% 可被后端解析、可直接保存
2. **简洁**：去掉自主创建接口、去掉复杂多轮确认
3. **统一**：Chroma 单库存储，元数据实现项目隔离
4. **清晰**：前端正确预览 → 编辑 → 保存(用户点击保存方能入库）
5. **高效**：RAG 检索精准、快速、安全
6. **可维护**：模块边界清晰、异常可捕获、可重试

## 3. 整体架构（重构后）

### 3.1 模块划分

1. **Vue 前端**
    - 对话展示
    - 用例生成结果预览与编辑
    - localStorage 维护对话历史
    - 无对话 ID
    
2. **Java 后端（核心入口）**
    - 统一鉴权、权限、项目隔离
    - 接口信息查询
    - AI 请求转发
    - 数据格式校验
    - 最终持久化
    
3. **FastAPI AI 服务**
    - RAG 检索
    - 对话历史拼接（Memory）
    - 用例结构化生成
    - Pydantic 格式强校验
    
4. **PyTest 执行引擎**
    - 仅负责用例执行，与 AI 无关
    
5. **Chroma 向量库**
    - 单库、单集合
    - 元数据：project_id /doc_id/doc_type /doc_name
    
6. **MySQL**
    - 业务数据
    - 不存对话历史
### 3.2 数据流总原则
- **Java 掌控所有业务数据**
- **AI 只做理解与结构化输出**
- **AI 不写库、不删库、不自主调用业务接口**
- **用例生成 = 已有接口组装**

## 4. RAG 模块重构方案

### 4.1 向量库结构
- **仅一个 Chroma 文件夹**
- **一个 Collection 存放所有项目数据**

### 4.2 元数据规范（必带）(示例)
```json
{
  "project_id": 1001,
  "doc_id": 567,
  "doc_type": "接口文档/用例文档/需求文档",
  "doc_name": "登录接口说明"
}
```

### 4.3 RAG 三大操作

#### （1）添加文档
1. 前端上传 → Java 保存 MySQL
2. Java 调用 FastAPI：/ai/rag/add
3. 参数：project_id、doc_id、doc_type、doc_name、content等等
4. AI 服务：
    - 章节切片
    - Embedding 向量化
    - 带元数据存入 Chroma
#### （2）检索问答
1. 前端传：project_id、messages（全量历史）、question
2. Java 直接转发 AI
3. AI 检索
4. 拼接上下文 → 大模型回答 → 返回前端

#### （3）删除文档
1. 前端删除 → Java 删除 MySQL
2. Java 调用 AI：/ai/rag/delete
3. 参数：project_id、doc_id
4. AI 执行：
```python
    collection.delete(where={"$and": [{"project_id": project_id}, {"doc_id": doc_id}]})
```


### 4.4 RAG 质量保证
- 按接口级切片
- 先关键词召回 + 向量检索
- 回答优先依据当前项目文档或用户上下文
- 无结果时按意图分流：项目/私有问题先明确“未检索到证据”再给排查建议；公开常识问题可直接正常回答
## 5. AI Agent 用例生成重构方案

### 5.0 核心规则

1. **用例生成核心架构**：前端 → 只传用户意图 + 项目 ID → AI 服务端 → ReAct 决策 → 调用工具拉接口元数据 → 再生成用例
2. **禁止 AI 自主创建接口**
3. **Java 先查询本项目所有接口，分析依赖**
4. **AI 只做：接口筛选 → 排序 → 组装用例**
5. **输出严格遵循后端 JSON Schema**

### 5.1 四个核心工具

1. **get_api_list(projectId)**
    拉该项目下全部接口清单（id、name、路径、方法、描述等等）
2. **get_api_detail(apiName/apiId)**
    拉单个接口完整契约：参数、校验、Header、响应、错误码
3. **get_api_relation(apiName)**
    查接口依赖，通过查询的接口清单，分析依赖关系（示例：登录 → 获取用户信息 → 下单）
4. **generate_testcase(api_detail, relation)**
    真正生成用例（也可以让 LLM 直接生成，不用单独工具）
### 5.2 用例生成核心流程（示例）

1. 前端输入：生成登录场景用例
2. 前端携带：project_id、user_id、prompt、messages（历史）
3. 后端接收后转ai服务
4. ai，生成用例
5. AI 服务：
    - 查询全量api_list，分析依赖
    - Memory 拼接历史
    - 加载用例 JSON Schema
    - 生成结构化用例
    - Pydantic 强校验
    - 失败自动重试 1 次
6. 返回标准用例 JSON 给 Java
7. Java **不保存**，直接返回前端
8. 前端支持同时支持Json二次编辑和接口新增组件的渲染填充编辑
9. 用户点击**保存** → Java 正式入库

### 5.3 用例 JSON 结构（前后端统一）(仅供示例，严格依据后端用例Schema)

```json
{
  "case_name": "登录正常流程",
  "project_id": 1001,
  "api_list": [
    {
      "api_id": 123,
      "extract": [{"key": "token", "path": "$.data.token"}],
      "assert": [{"field": "$.code", "op": "==", "expect": 200}]
    }
  ]
}
```

### 5.4 Pydantic 校验规则

- 校验符合预定义的后端新增用例的Schema
- 接口列表非空

## 6. 对话历史模块重构

### 6.1 方案

- **删除对话 ID（conversation_id）**
- **前端 localStorage 维护完整历史**
- **每次请求携带全量 messages 数组**
- Java 不存储对话
- AI Memory 仅做拼接，不存储

### 6.2 前端传递格式（示例参考）

```json
{
  "project_id": xxx,
  "user_id":xxx,
  "messages": [
    {"role": "user", "content": "xxx"},
    {"role": "assistant", "content": "yyy"}
  ],
  "question": "生成登录用例"
}
```

### 6.3 Memory 作用
- 把 messages 拼接为 prompt
- 无存储、无状态、无并发问题

## 7. 前后端交互流程（参考）

### 7.1 RAG 问答流程

1. 前端 localStorage 历史 → 打包
2. 前端 → Java：/ai/chat
3. Java → FastAPI：/ai/chat
4. AI 检索本项目文档
5. 回答返回前端
6. 前端追加历史到 localStorage

### 7.2 用例生成流程

1. 前端输入指令
2. 前端 → Java →FastAPI ai服务
3. **ReAct架构**：思考 → 调用工具查已有接口列表 → 观察结果 → 决策生成 / 提示不存在
4. 若相关接口存在，AI 生成用例 → 校验
5. 返回前端预览
6. 编辑 → 保存 → 入库

### 7.3 文档删除同步向量流程

1. 前端删文档 → Java
2. Java 删 MySQL
3. Java → AI：/ai/rag/delete(携带project_id、doc_id）
4. AI 按元数据删除向量

## 8. 异常处理机制（参考依据）

### 8.1 RAG 异常

- 无相关文档（项目/私有问题）→ 先说明“当前项目未检索到相关证据”，再补充可执行排查建议并询问补充信息
- 无相关文档（公开常识问题）→ 直接按通用知识回答，不机械返回“未找到相关知识”
- 向量库异常 → 优先回退通用回答，并提示稍后可重试知识库检索

### 8.2 用例生成异常

- 无接口/无权限/详情读取失败 → 返回具体原因，不统一为“请先创建接口”
- AI 格式错误 → 自动重试 1 次
- 重试失败 → 返回：“用例生成失败，请更换描述”

### 8.3 前端异常

- 加载失败 → 支持重新生成
- 不自动保存、不产生脏数据

## 9. 约束与边界

1. AI 不能修改任何业务数据
2. AI 不能创建 / 删除接口
3. 用例仅可使用已存在的 api_id
4. 所有跨项目数据严格隔离
5. 对话历史不上报后端
6. 生成≠保存，保存必须用户主动触发

## 10. 验证标准

1. RAG 问答不会跨项目泄露
2. 用例生成 100% 合法可保存
3. 删除文档同步删除向量
4. 多用户并发不串数据
5. 前端预览正常、编辑正常、保存正常
6. 无崩溃、无格式报错
